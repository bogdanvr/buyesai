# CRM Communications Staging Checklist

## Цель

Этот чек-лист закрывает приемку модуля `crm_communications` на staging перед включением в production.

Проверяем:
- Telegram inbound / outbound
- Email inbound / outbound
- ambiguous binding
- retry flow
- manual recovery
- отображение переписки в карточках компании и сделки

## Предусловия

- применены миграции;
- заполнены `IMAP_*` настройки;
- настроен webhook Telegram;
- доступен исходящий SMTP;
- в CRM есть:
  - компания;
  - контакт с email;
  - контакт с telegram;
  - сделка;
  - сценарий с двумя активными сделками одной компании.

## Автоматические проверки

Запустить:

```bash
python manage.py check
python manage.py test crm_communications
```

Ожидаемо:
- оба запуска проходят без ошибок;
- `crm_communications` зелёный полностью.

## Telegram QA

### 1. Входящее сообщение Telegram

Шаги:
1. Отправить личное сообщение в Telegram бота.
2. Убедиться, что webhook отработал.
3. Открыть компанию и сделку в CRM.

Ожидаемо:
- создался `Conversation`;
- создался `Message`;
- создался `Touch`;
- сообщение видно в панели переписки;
- из ленты событий есть переход в переписку.

### 2. Исходящее сообщение Telegram

Шаги:
1. Открыть conversation.
2. Отправить сообщение из CRM.

Ожидаемо:
- сообщение уходит в Telegram;
- статус сообщения `sent`;
- есть `MessageAttemptLog`;
- в сделке/компании появляется `Touch`.

### 3. Telegram без корректного chat_id

Шаги:
1. У контакта оставить `telegram`, который не является numeric `chat_id`.
2. Отправить сообщение из CRM.

Ожидаемо:
- сообщение попадает в `requires_manual_retry`;
- создаётся запись в `DeliveryFailureQueue`.

## Email QA

### 4. Входящее письмо

Шаги:
1. Отправить письмо на ящик CRM.
2. Выполнить:

```bash
python manage.py fetch_imap_inbox
```

Ожидаемо:
- письмо появляется в `Conversation`;
- создаётся `Message`;
- создаётся `Touch`;
- вложения сохраняются как `MessageAttachment`.

### 5. Ответ в том же thread

Шаги:
1. Ответить на то же письмо.
2. Повторно выполнить `fetch_imap_inbox`.

Ожидаемо:
- второе письмо попадает в тот же `Conversation`;
- новый диалог не создаётся.

### 6. Исходящее письмо

Шаги:
1. Открыть conversation по email.
2. Отправить ответ из CRM.

Ожидаемо:
- письмо уходит через SMTP;
- заполняются `Message-ID`, `In-Reply-To`, `References`;
- появляется успешный `MessageAttemptLog`;
- создаётся communication `Touch`.

## Ambiguous Binding QA

### 7. Две активные сделки у одной компании

Шаги:
1. Создать у одной компании две активные сделки.
2. Прислать новое письмо или Telegram сообщение от этого контакта.

Ожидаемо:
- `Conversation.requires_manual_binding = true`;
- у conversation нет авто-привязки к сделке;
- в карточке сделки/компании виден alert ручной привязки.

### 8. Ручная привязка

Шаги:
1. Выполнить bind через UI или API.

Ожидаемо:
- `Conversation` получает правильную сделку;
- последующие сообщения идут в тот же диалог;
- при bind failure item обновляются также `Message` и `Touch`.

## Retry QA

### 9. Ошибка доставки

Шаги:
1. Спровоцировать failure для исходящего email или telegram.
2. Открыть `DeliveryFailureQueue`.

Ожидаемо:
- есть failure item;
- видны причина и последний attempt.

### 10. Ручной retry

Шаги:
1. Исправить получателя.
2. Выполнить retry из API/UI.

Ожидаемо:
- сообщение отправляется;
- failure item уходит в `resolved`;
- в логах появляется новая попытка.

## Manual Recovery QA

### 11. Resolve без повтора

Шаги:
1. Открыть failure item.
2. Выполнить `resolve` с комментарием.

Ожидаемо:
- статус `resolved`;
- комментарий сохранён;
- назначен ответственный.

### 12. Close кейса

Шаги:
1. Выполнить `close` с комментарием.

Ожидаемо:
- статус `closed`;
- элемент пропадает из default списка failures.

## UI QA

### 13. Компания

Проверить:
- виден список conversations компании;
- открывается thread;
- composer отправляет сообщение;
- polling подтягивает новые сообщения.

### 14. Сделка

Проверить:
- в сделке видны только связанные conversations;
- если есть непривязанный диалог компании, виден alert;
- bind из сделки привязывает conversation к текущей сделке.

### 15. Timeline / Touch integration

Проверить:
- из `Touch` в ленте есть переход в переписку;
- открывается нужный conversation;
- и входящие, и исходящие сообщения создают `Touch`.

## Финальное решение о готовности

Считать staging пройденным, если:
- автоматические тесты зелёные;
- все 15 сценариев выше проходят;
- не осталось `MessageWebhookEvent` в `failed`, которые связаны с тестовым контуром;
- очередь `DeliveryFailureQueue` после тестов либо пуста, либо содержит только ожидаемые ручные кейсы.
